# Implementation Plan: Smart Task Reminder System with Responsive Card UI

**Branch**: `004-smart-task-reminders` | **Date**: 2026-05-19 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/004-smart-task-reminders/spec.md`

## Summary

Add a browser-based task reminder system to the existing NestJS + PostgreSQL + Next.js todo application. Users can assign a `reminderTime` to any task; a 30-second frontend poller triggers desktop notifications (via a minimal service worker with BroadcastChannel) and an alarm sound at the configured time. Users can snooze (5-minute re-trigger) or dismiss. All reminder state persists in PostgreSQL via 5 new Prisma-managed fields. Simultaneously, the existing flex-list task layout is upgraded to a responsive CSS grid (1 ‚Üí 2 ‚Üí 3/4 columns by breakpoint) with `rectSortingStrategy` for drag-and-drop.

## Key Design Decisions

### 1. Service Worker for Notification Actions

**Decision**: `public/sw.js` (‚â§30 lines) listens for `notificationclick`, posts `{ action, taskId }` to `BroadcastChannel('task-reminders')`. Page listens and calls API.

**Why**: `new Notification()` ignores `actions` on all browsers ‚Äî SW is mandatory for action buttons. BroadcastChannel is simpler than `clients.matchAll()`.

**Fallback**: Feature-detect `'actions' in Notification.prototype`; if false (Firefox, Safari iOS), use in-app banner only. In-app banner always rendered as a second layer regardless.

### 2. Audio Unlock Pattern

**Decision**: On first user interaction with the page, call `audio.play().then(() => audio.pause())` to unlock the audio context. Store the unlocked state in a module-level variable.

**Why**: Browsers block `audio.play()` without prior user interaction. Since reminders fire from a timer (not a click), unlocking proactively on first page interaction ensures alarm playback works.

### 3. snoozeUntil Computed Server-Side

**Decision**: Frontend sends `{ reminderStatus: "snoozed", notificationTriggered: false }`. Backend computes and sets `snoozeUntil = now + 5 minutes`.

**Why**: Prevents clock-skew issues if client and server clocks differ. Server is authoritative for all timestamps.

### 4. rectSortingStrategy for Grid DnD

**Decision**: Switch `DraggableTaskList.tsx` to `rectSortingStrategy` and change container from `flex flex-col gap-2` to `grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-4`.

**Why**: `verticalListSortingStrategy` is Y-axis only ‚Äî produces wrong sort order in multi-column grids. `rectSortingStrategy` uses full 2D bounding-rect math.

### 5. ReminderStatus State Machine (DB enum as strings)

**Decision**: `reminderStatus` stored as plain string with `@IsIn` validation. Values: `none`, `pending`, `snoozed`, `dismissed`. No Prisma native enum (avoids migration complexity for enum type addition).

**Triggered state** is derived client-side from: `notificationTriggered = true` + `reminderStatus = 'pending'` (or `snoozeUntil` past). Not stored in DB to keep the state machine simple.

### 6. Drag-and-Drop in Grid View Scope

**Decision**: Drag-and-drop remains functional in grid layout using `rectSortingStrategy`. It is disabled only when active sort (non-custom) or filters are applied ‚Äî same behavior as spec 003.
